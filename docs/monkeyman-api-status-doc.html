🌲 MonkeyMan Agent — API Status Document

Full inventory of every API the system touches, grouped by provider, with a checkable status column. Ready to paste into Google Docs.

1. What this document covers

The MonkeyMan Agent is a Node.js + Express service that lets employees chat with an LLM-backed assistant. The assistant talks to three kinds of APIs on the user's behalf:

  • Google Gemini — powers the reasoning and tool-use loop for every chat message.
  • MonkeyMan Backend at https://server.monkeymans.com/api — source of truth for employees, tasks, PTO, VTR, incidents, on-call, forms, permits.
  • Internal endpoints exposed by this service — login, session, chat, health.
How to use the table below: walk each row top to bottom. For every row, put a tick (✓) or "Y" in the Working column if that API behaves as expected, or in the Not Working column if it does not. Use the Notes column for details.

5. Notes for the tech team

Before running these tests: the system is mid-migration. Two changes are pending:
  1. Remove OTP — the Send_OTP / Verify_OTP tools, otp.js, and the /mm-send-otp + /mm-verify-otp routes are no longer needed. Login already proves identity.
  2. Swap Anthropic → Google Gemini — replace @anthropic-ai/sdk with @google/genai, swap ANTHROPIC_API_KEY for GEMINI_API_KEY, and rewrite the agent loop in agent.js to use Gemini's generateContent + function-calling. All 24 tools in tools.js port directly once input_schema is renamed to parameters.

How each status is decided

  • Working — API call returns the expected shape in under a reasonable timeout, and downstream consumers (UI / agent) handle the response correctly.
  • Not Working — call times out, returns a non-2xx status, returns malformed data, or downstream code errors when consuming the response. Add details in the Notes column.

Environment variables in use

  • GEMINI_API_KEY — LLM provider key (Gemini).
  • MONKEYMAN_API_BASE — defaults to https://server.monkeymans.com/api.
  • LOGIN_PASSWORD — shared password for all users (currently monkeyman2024).
  • PORT — HTTP port (default 3000).
  • MAX_TOOL_ROUNDS — cap on tool-use loops per user message (default 15).
